# 학교 부서별 업무 링크 정적 사이트 설계명세서

## 1. 개요

### 1.1 프로젝트명
학교 부서별 업무 링크 허브

### 1.2 목적
학교 내 여러 부서의 주요 업무 링크를 한 화면에서 직관적으로 확인하고 이동할 수 있는 정적 웹사이트를 구축한다. 메인 페이지는 마인드맵처럼 자유롭게 연결·배치되는 느낌을 주되, 실제 배치는 보이지 않는 그리드 체계를 기반으로 하여 정돈된 카드형 UI를 유지한다.

### 1.3 핵심 요구사항
- 메인 페이지에 교무부, 학생부, 진학부, 교육정보부, 행정실 등 학교 부서별 업무 카드를 표시한다.
- 각 카드는 부서명, 담당업무 한 줄 설명, 바로가기 링크를 포함한다.
- 카드들은 마인드맵처럼 연결선으로 관계를 표현할 수 있어야 한다.
- 사용자는 카드를 자유롭게 배치할 수 있으나, 내부적으로는 보이지 않는 그리드에 스냅되어 깔끔하게 정렬된다.
- 카드 작성·수정·삭제는 로그인한 사용자만 가능하다.
- 테스트 단계에서는 교사명 `Master`, 비밀번호 `2026`으로 로그인한다.
- 향후에는 교사명·비밀번호 DB를 참고하는 인증 구조로 확장한다.
- 모바일에서도 카드와 연결 구조가 보기 좋게 반응형으로 동작해야 한다.

## 2. 기술 스택

| 구분 | 기술 | 설명 |
|---|---|---|
| Framework | Next.js | 정적 사이트 생성 및 React 기반 UI 구성 |
| Language | TypeScript | 카드·인증·레이아웃 데이터 타입 안정성 확보 |

## 5. 레이아웃 설계

### 5.1 보이지 않는 그리드
- 보드 내부를 `gridSize = 160px` 단위로 관리한다.
- 카드를 드래그하면 실제 좌표는 자유롭게 움직이는 것처럼 보이지만, 드롭 시 가장 가까운 그리드 좌표로 스냅한다.
- 저장되는 위치값은 픽셀값이 아니라 `{ x: number, y: number }` 형태의 그리드 좌표로 저장한다.

```ts
const GRID_SIZE = 160;

export function snapToGrid(pixelX: number, pixelY: number) {
  return {
    x: Math.round(pixelX / GRID_SIZE),
    y: Math.round(pixelY / GRID_SIZE),
  };
}

export function gridToPixel(x: number, y: number) {
  return {
    left: x * GRID_SIZE,
    top: y * GRID_SIZE,
  };
}
```

### 5.2 카드 연결선
- SVG 레이어를 카드 레이어 아래에 배치한다.
- 각 카드의 중심점을 계산하여 `line` 또는 `path`로 연결한다.
- 연결선은 회색 계열의 얇은 선을 사용하고, 모바일에서는 과도한 복잡도를 줄이기 위해 투명도를 낮춘다.

### 5.3 모바일 반응형
- 화면 폭 768px 미만에서는 카드가 1열 또는 2열로 자동 재배치된다.
- 모바일에서는 드래그 배치보다 카드 목록 탐색성을 우선한다.
- 연결선은 유지하되, 복잡도가 높아지면 `연결 보기` 토글을 제공한다.

### 5.4 API 동기화 및 캐싱 정책
- **시간 기반 캐싱 (조회 성능 최적화)**:
  - 구글 시트 API의 일시적인 동시 요청 차단(Rate Limit)을 예방하고 불필요한 트래픽을 아끼기 위해, 5분(300,000ms) 시간 기반 캐싱 정책을 적용한다.
  - 마운트 또는 단순 새로고침 시 캐시 만료 전까지는 구글 시트를 다시 로드하지 않고 브라우저에 캐싱된 카드 상태를 즉시 활용하여 렌더링 속도를 극대화한다.
- **편집 시 즉시 동기화 및 캐시 갱신**:
  - 로그인한 사용자가 카드를 추가/수정/삭제하거나 위치 배치를 변경하고 '위치 저장'을 실행하는 경우에는 구글 시트 원격 데이터베이스에 변경 사항을 즉시 반영(POST)한다.
  - 저장 작업이 성공적으로 완료되면 즉시 브라우저의 캐시 타임스탬프(`school-cards-cache-time`)를 현재 시각으로 갱신하여 정합성을 유지한다.

### 5.5 데이터 강건성 및 예외/복구 처리
- **좌표(position) 유효성 검증 및 복구**:
  - 구글 시트 API로부터 로드한 노드의 `position` 값이 없거나, `x === 0 && y === 0` 등 비정상적인 값일 경우, `INITIAL_CARDS`에 정의된 기본 좌표로 자동 복원(Fallback)한다.
  - `INITIAL_CARDS`에 정의되지 않은 새로 생성된 커스텀 부서 카드는 중앙 노드(`152, 116`) 기준 사방 오프셋을 동적으로 계산하여 포개지지 않도록 분산 스폰한다.
- **그리드 스케일 자동 마이그레이션**:
  - 구글 시트 상에 저장된 데이터의 스케일이 구버전(40px 그리드 스케일, 중앙 노드 x < 50)이거나 2세대 중앙 시프트 이전 스케일(x = 96, y = 72)인 경우, 최신 10px 그리드 체계로 자동 변환하여 위치 왜곡을 방지한다.
- **부서 연결선(parentId) 정제 예외 처리**:
  - 1단계 부서 노드(`type === 'department'`)의 `parentId`가 누락되거나 잘못된 값으로 왔을 경우, 중앙 학교 노드(`'center-gwangyang'`)로 강제 매핑 및 복구하여 연결선이 짤리는 현상을 원천 방지한다.

테스트 단계에서는 아래 계정을 사용한다.

| 교사명 | 비밀번호 |
|---|---|
| Master | 2026 |

### 6.2 인증 흐름
1. 사용자가 로그인 버튼 클릭
2. 로그인 모달에서 교사명·비밀번호 입력
3. 입력값이 `Master / 2026`과 일치하면 로그인 성공
4. 로그인 성공 시 카드 작성·수정·삭제 기능 활성화
5. 로그아웃 시 읽기 전용 모드로 전환

```ts
export type Teacher = {

};

  // 운영 단계: DB에서 교사명 조회 후 passwordHash 검증
}
```

### 6.4 보안 주의
- 테스트 비밀번호 `2026`은 운영 배포 전 반드시 제거한다.
- 운영 단계에서는 평문 비밀번호 저장을 금지하고, 해시 저장을 적용한다.
- 정적 사이트만으로는 안전한 서버 인증이 어렵기 때문에 운영 인증은 Next.js Route Handler 또는 별도 백엔드/API 연동을 전제로 한다.

## 10. UI 디자인 방향

## 12. 수용 기준

- [ ] 메인 페이지에 최소 5개 부서 카드가 표시된다.
- [ ] 각 카드에는 부서명, 담당업무 한 줄 설명, 바로가기 링크가 있다.
- [ ] 카드 간 연결선이 표시된다.

- [ ] 로그인하지 않은 사용자는 카드 작성·수정·삭제 버튼을 볼 수 없다.
- [ ] `Master / 2026`으로 로그인하면 카드 작성·수정·삭제 기능이 활성화된다.
- [ ] 모바일 화면에서도 카드 정보가 잘 보인다.
- [ ] 운영 단계 DB 인증으로 확장할 수 있도록 인증 로직이 분리되어 있다.
